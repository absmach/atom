This walkthrough builds one complete access-control scenario using only the Atom Next UI â€”
no `curl`, no GraphQL. By the end you will have:

- a tenant (`acme-corp`),
- two entities â€” a human (`Alice`) and a service (`billing-service`),
- a resource (`invoice-events`) two actions apply to,
- two permission blocks (read-only, and read+write),
- two roles that wrap those blocks,
- two direct policies that grant the blocks to Alice and billing-service, and
- four authorization checks in the debugger, including one **allow** and one **deny**.

It mirrors the [Getting Started with Atom](https://www.absmach.eu/blog/getting-started-with-atom/)
blog post. See [Access Control](/access-control) for the concepts behind roles, permission
blocks, and direct policies.

## Prerequisites

Atom is running (`make up`) and reachable at `http://localhost:8080` (API) and
`http://localhost:3005` (UI). See [Quickstart](/quickstart) if you have not started the stack
yet.

<Steps>
<Step>

### Sign in

Open `http://localhost:3005/login` and sign in with `admin` / `12345678` (the default dev
credentials â€” change `ADMIN_SECRET` in `.env` before running this anywhere else). You land on
the **Dashboard**, which summarizes tenants, entities, roles, and platform activity.

![Sign in and dashboard](/img/user-guide/dashboard/dashboard-overview.png)

</Step>
<Step>

### Create a tenant

Open **Tenants** in the sidebar. The table is empty on a fresh instance. Click **+ Create** in
the top-right corner.

![Tenants page, empty, with the Create button highlighted](/img/user-guide/tenants/tenants-list-create-highlight.png)

Fill in:

- **Name**: `acme-corp`
- **Alias**: `acme`

Leave **Tags** and **Attributes JSON** empty, then click **Save tenant**.

![Create tenant dialog filled in](/img/user-guide/tenants/tenants-create-dialog.png)

The new tenant appears in the table with row actions **Inspect**, **Edit**, **Freeze**,
**Disable**, and **Delete**. Click **Inspect** if you want to copy its ID â€” you won't need it
for this walkthrough since every later form lets you pick `acme-corp` by name.

![Tenants table with acme-corp](/img/user-guide/tenants/tenants-list-populated.png)

Switch your working context to the new tenant using the **Global / platform** switcher at the
top of the sidebar, or leave it on **Global** and pick `acme-corp` explicitly in each form
below â€” both work, since every create dialog has its own **Tenant** field.

</Step>
<Step>

### Create two entities

Open **Entities**. Click **+ Create**.

![Entities page with Create button highlighted](/img/user-guide/entities/entities-list-create-highlight.png)

Create the first entity:

- **Name**: `Alice`
- **Kind**: `human`
- **Tenant**: `acme-corp`
- Leave **Profile** as **No profile**.

![Create entity dialog](/img/user-guide/entities/entities-create-dialog.png)

Click **Save entity**. Repeat for a second entity:

- **Name**: `billing-service`
- **Kind**: `service` (the **Kind** dropdown also offers `device`, `workload`, and
  `application` â€” pick whichever matches what you're modeling)
- **Tenant**: `acme-corp`

![Entity kind options](/img/user-guide/entities/entities-kind-options.png)

Both entities now appear in the table, alongside any entities Atom seeded on first boot
(`admin`, `example-service`).

![Entities table populated](/img/user-guide/entities/entities-list-populated.png)

</Step>
<Step>

### Create a resource

Open **Resources**. The table starts empty. Click **+ Create**.

![Resources page, empty](/img/user-guide/resources/resources-list-empty.png)

Fill in:

- **Kind**: `invoice`
- **Name**: `invoice-events`
- **Tenant**: `acme-corp`
- Leave **Owner entity** and **Attributes JSON** at their defaults.

![Create resource dialog filled in](/img/user-guide/resources/resources-create-dialog.png)

Click **Create resource**. Open the new row's **Inspect** view and copy the **ID** â€” you'll
paste it into the permission block scope in a later step.

![Resource inspect view with ID](/img/user-guide/resources/resource-inspect.png)

</Step>
<Step>

### Tell Atom which actions apply to invoices

Actions like `read` and `write` already exist platform-wide (Atom seeds a standard catalog),
but Atom still needs to know that `read` and `write` are valid for the `invoice` resource
kind. Open **Actions â†’ Action Applicability** and click **+ Create**.

![Action Applicability create dialog](/img/user-guide/actions/action-applicability-create-dialog.png)

Create one row for read:

- **action_id**: `read`
- **object_kind**: `resource`
- **object_type**: `resource:invoice`

![Action Applicability filled in for resource:invoice](/img/user-guide/actions/action-applicability-create-filled.png)

Click **Create row**, then repeat with **action_id** set to `write` (same **object_kind** and
**object_type**). See [Actions](/user-guide/actions) for what **Assignment Guardrails** (the
third tab in this section) is for â€” you don't need it here.

</Step>
<Step>

### Create two permission blocks

Open **Permission Blocks** and click **+ Create**. This is a 5-step wizard.

**Block 1 â€” read + write.**

1. **Boundary** â€” set **Tenant boundary** to `acme-corp`, leave **Effect** as `allow`.
2. **Scope** â€” set **Scope mode** to `Exact object`, **Object kind** to `Resource`, enter
   **Object type** `resource:invoice`, and paste the `invoice-events` resource ID into
   **Exact object ID**.
3. **Actions** â€” check both `read` and `write`.
4. **Conditions** â€” leave the JSON as `{}` (no ABAC conditions for this walkthrough).
5. **Review** â€” confirm the summary, then click **Create permission block**.

![Permission block wizard, scope step filled in](/img/user-guide/permission-blocks/pb-wizard-step2-scope-filled.png)

![Permission block wizard, actions step](/img/user-guide/permission-blocks/pb-wizard-step3-actions.png)

**Block 2 â€” read only.** Repeat the wizard with the same tenant, scope, and object ID, but
check only `read` in the **Actions** step.

Both blocks now appear in the table, alongside the seeded platform and tenant-admin blocks.

![Permission blocks table populated](/img/user-guide/permission-blocks/permission-blocks-list-populated.png)

</Step>
<Step>

### Create two roles

Open **Roles** and click **+ Create**. This is a 3-step wizard: **Basics**, **Permission
blocks**, **Review**.

Create **invoices-reader**:

1. **Tenant**: `acme-corp`. **Role name**: `invoices-reader`.
2. Under **Permission blocks**, use the **Add permission block** dropdown and pick the
   read-only block (`resource ... Â· allow Â· read`).
3. Review, then **Create role**.

![Role wizard, permission block selected](/img/user-guide/roles/role-wizard-step2-selected.png)

Create **invoices-reader-writer** the same way, attaching the read+write block instead.

![Roles table populated](/img/user-guide/roles/roles-list-populated.png)

<Callout type="info">
These roles exist to mirror the permission blocks by name and aren't attached to Alice or
billing-service directly in this walkthrough â€” the next step grants access with **Direct
Policies** instead, which is what the debugger will explain. Atom's UI does not yet expose a
dedicated "assign role to entity" screen; role-to-principal-group bindings you see in the
Authorization debugger's explanation (like `domain-creator` via `authenticated-users`) are
seeded through migrations today. See [Roles](/user-guide/roles) for more.
</Callout>

</Step>
<Step>

### Grant access with direct policies

Open **Direct Policies** and click **+ Create**. This is a 4-step wizard: **Tenant**,
**Subject**, **Permission block**, **Review**.

**Alice's policy** â€” read-only:

1. **Tenant boundary**: `acme-corp`.
2. **Subject kind**: `Entity`. **Subject**: `Alice (human)`.
3. **Permission block**: the read-only block.
4. Review, then **Create policy**.

**billing-service's policy** â€” read+write: repeat, picking `billing-service (service)` as the
subject and the read+write block.

![Direct policies table populated](/img/user-guide/direct-policies/direct-policies-list-populated.png)

</Step>
<Step>

### Check authorization

Open **Authorization**. This is the authorization debugger â€” pick **Who**, **Can do**,
**Target type**, and **Resource**, then click **Explain decision**.

![Authorization debugger, empty, with Explain decision highlighted](/img/user-guide/authorization/authz-explain-highlight.png)

Run all four combinations from this scenario:

| Who | Can do | Target | Expected |
| --- | --- | --- | --- |
| `billing-service` | `read` | `invoice-events` | **Allowed** |
| `billing-service` | `write` | `invoice-events` | **Allowed** |
| `Alice` | `read` | `invoice-events` | **Allowed** |
| `Alice` | `write` | `invoice-events` | **Denied** |

For `billing-service` reading, the decision panel shows **Allowed**, the matched permission
(`effect: Allow`, `action: read`, scope `exact object <resource id>`), and â€” because the same
block also grants `write` â€” a second matched permission for that action too.

![Allowed decision with matched-permission detail](/img/user-guide/authorization/authz-decision-allowed.png)

For `Alice` writing, the decision is **Denied** with reason `no matching allow policy`. The
panel also shows *why* it isn't a coincidence: Alice's read-only permission block is
**Skipped â€” Action not covered by this grant**, and it even surfaces an unrelated inherited
role (`domain-creator`, via the seeded `authenticated-users` principal group) that also
doesn't cover `write`.

![Denied decision with skipped-permission detail](/img/user-guide/authorization/authz-decision-denied.png)

</Step>
</Steps>

## What you built

- A tenant boundary (`acme-corp`) that scopes everything else.
- Two entities representing a human and a service.
- A resource and the action applicability rows that make `read`/`write` valid for it.
- Two permission blocks expressing "read-only" and "read+write" as reusable, named grants.
- Two roles wrapping those blocks (for later reuse â€” see [Roles](/user-guide/roles)).
- Two direct policies binding the blocks straight to Alice and billing-service.
- A debugged, explainable authorization decision for every combination.

Continue to [Entities](/user-guide/entities) for credentials (passwords, API keys,
certificates), or [Audit](/user-guide/audit) to see every action above recorded as an
immutable audit log entry â€” including the `authz.explain` deny you just produced.
